## TASK 1: Architecture Review (in plan)

See dependency map above. Key findings:

1. **AttackEngine interfaces exist but are UNUSED.** `AttackPlanner`, `AttackExecutor`, `AttackVerifier` are defined in `internal/attack/engine.go` but never referenced by the orchestrator.

2. **The `ModuleAuthDiscovery` case in orchestrator.go already proves the browser-agent pattern works.** It uses: Browser Context → AI Decision → Browser Action → Result → History. This exact pattern must be generalized for attack execution.

3. **The AI Hypothesis execution (line 676-754) partially demonstrates the agent pattern** but still uses `net/http` instead of Playwright.

## TASK 5: Session Journal Design

**New file: `internal/attack/journal.go`**

```go
package attack

import "time"

// JournalEntry records a single step in a multi-step attack workflow.
type JournalEntry struct {
    Timestamp time.Time     `json:"timestamp"`
    Step      int           `json:"step"`
    Goal      string        `json:"goal"`       // e.g. "Find login form"
    Action    string        `json:"action"`      // e.g. "navigate", "fill", "click"
    Selector  string        `json:"selector,omitempty"`
    Value     string        `json:"value,omitempty"`
    Reasoning string        `json:"reasoning"`   // AI's reasoning for this action
    Result    *ActionResult `json:"result"`
    Verdict   AttackVerdict `json:"verdict,omitempty"` // CONFIRMED, FAILED, etc.
}

// SessionJournal provides persistent attack memory.
// Kept in-memory during scan and persisted to SQLite.
type SessionJournal struct {
    ScanID     string          `json:"scan_id"`
    Entries    []JournalEntry  `json:"entries"`
    CurrentStep int           `json:"current_step"`
}
```

Storage persistence in `internal/storage/journal.go`:
```sql
CREATE TABLE IF NOT EXISTS attack_journal (
    id        TEXT PRIMARY KEY,
    scan_id   TEXT NOT NULL,
    step      INTEGER NOT NULL,
    goal      TEXT NOT NULL,
    action    TEXT NOT NULL,
    selector  TEXT,
    value     TEXT,
    reasoning TEXT,
    verdict   TEXT,
    evidence  TEXT, -- JSON
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (scan_id) REFERENCES scans(id) ON DELETE CASCADE
);
```

The AI receives the full journal on every decision call so it can reason about multi-step attacks like:
- Register → Login → Change Role → Access Admin

## TASK 6: SQL Injection Refactor Plan

### Current: `internal/scanner/sqli.go`

```
SQLi Scanner (Go)
  │
  ├─ ExtractInsertionPoints (parse URL params, form fields, JSON body)
  ├─ fetchBaseline → as.client.Do (net/http)
  ├─ checkErrorBased → as.client.Do (net/http) — error markers
  ├─ checkBooleanBased → as.client.Do (net/http) — response length diff
  └─ checkTimeBased → as.client.Do (net/http) — timing delta
       │
       ▼
  SaveFindingWithEvidence
```

### Desired: Agent-Driven SQLi

```
AI Agent
  │
  ├── 1. Hypothesize: "Parameter 'id' may be SQL injectable"
  │    (based on page context: URL has query params, form has text inputs)
  │
  ├── 2. Plan Attack Strategy:
  │    - Choose payload class (error-based, boolean, time-based, UNION)
  │    - Select specific payloads
  │    - Determine injection point (query param, form field, etc.)
  │
  ├── 3. Browser Attack Loop (for each payload):
  │    │
  │    ├── Navigate to target page
  │    ├── Fill form fields with payload (if form-based)
  │    ├── Click submit / trigger request
  │    ├── Wait for response / observe page change
  │    ├── Capture: URL, page source, screenshot, network events
  │    │
  │    └── ActionResult → AI decides next payload or declares success/failure
  │
  ├── 4. Verify Result:
  │    - Error-based: DB error visible in page source?
  │    - Boolean-based: page content differs from baseline?
  │    - Time-based: response time > threshold?
  │    - UNION-based: unexpected data in response?
  │
  ├── 5. Capture Evidence:
  │    - Screenshot showing DB error or data
  │    - HTTP flow (request + response)
  │    - AI reasoning narrative
  │
  └── 6. Create Finding:
     - Title: "SQL Injection in Parameter: [name]"
     - Evidence: flow ID, screenshot, AI explanation
     - Severity: CRITICAL (verified) / HIGH (suspected)
```

### SQLi Agent Architecture

```go
// internal/attack/sql_agent.go

// AgentSQLiExecutor runs SQL injection through the browser using AI decision-making.
type AgentSQLiExecutor struct {
    db           *storage.DB
    browser      *browser.Client
    aiClient     aiv1connect.AiServiceClient
    sessionID    string
    scanID       string
    proxyPort    int
    journal      *SessionJournal
}

// Execute runs the complete agent-driven SQLi workflow.
func (e *AgentSQLiExecutor) Execute(ctx context.Context, target browser.BrowserContext) error {
    // 1. Ask AI to analyze the page and decide attack strategy
    strategy, err := e.aiClient.PlanSQLiAttack(ctx, &PlanSQLiInput{
        CurrentContext: target,
        Endpoint:       target.CurrentURL,
        Forms:          target.Forms,
        Parameters:     extractParams(target),
    })
    
    // 2. For each strategy (error, boolean, time, union), execute via browser
    for _, payload := range strategy.Payloads {
        result := e.executePayloadViaBrowser(ctx, payload, target)
        e.journal.AddEntry(JournalEntry{
            Action:    "sqli_test",
            Value:     payload.Value,
            Reasoning: strategy.Reasoning,
            Result:    result,
        })
        
        // 3. Ask AI to verify the result
        verdict, _ := e.aiClient.VerifyAttackResult(ctx, &VerifyInput{
            Payload:  payload.Value,
            Response: result,
        })
        
        if verdict.Confirmed {
            return e.saveFinding(ctx, payload, result, verdict)
        }
    }
    return nil
}
```

This mirrors the already-working `ModuleAuthDiscovery` pattern but generalizes it to attack execution.
