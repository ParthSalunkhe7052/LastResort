# Research: Fix Duplicate SQLi Execution

**Date**: 2026-06-06

## 1. Executive Summary
The goal is to consolidate SQL injection scanning into a single `sqli_agent` module, avoiding duplicate execution and findings from `sqli_basic`. Research shows that `sqli_basic` has already been removed from `ProfileModules` in `phase.go` and is not handled in `orchestrator.go`. The `sqli_agent` module in `internal/attack/sql_agent.go` already incorporates static payloads as its first phase. However, `internal/orchestrator/verification_engine.go` fails to compile due to a missing import of the `scanner` package, which is used for deterministic verification.

## 2. Technical Context
- **Affected Components**:
    - `internal/orchestrator/phase.go`: Defines scan profiles and module constants.
    - `internal/orchestrator/orchestrator.go`: Manages scan execution and browser health checks.
    - `internal/orchestrator/verification_engine.go`: Handles vulnerability verification logic.
    - `internal/attack/sql_agent.go`: Implements the SQLi agent attack logic.
    - `internal/scanner/sql_verifier.go`: Contains deterministic SQLi verification rules.

- **Current Behavior**:
    - `ProfileModules` in `phase.go` correctly lists 13 modules for STANDARD and 14 for DEEP, including `sqli_agent` but not `sqli_basic`.
    - `sqli_agent` implementation in `sql_agent.go` already includes a loop over `scanner.SQLiPayloads`.
    - `Orchestrator.Start` already includes a browser health check using `o.browserClient.IsOnline`.
    - `verification_engine.go` attempts to call `scanner.VerifySQLiDeterministic` but does not import the `scanner` package.

## 3. Findings & Analysis
- **Module Consolidation**: The consolidation of `sqli_basic` into `sqli_agent` appears to be mostly complete in terms of logic and profile configuration. `ModuleSqliBasic` is still defined as a constant in `phase.go` but is unused.
- **Build Failure**: `internal/orchestrator/verification_engine.go:175:9: undefined: scanner`. This is a direct consequence of missing `github.com/parth/lastresort/internal/scanner` in the imports.
- **Circular Dependency Check**: `internal/orchestrator` imports `internal/scanner`. `internal/scanner` does not import `internal/orchestrator`. Thus, adding the import to `verification_engine.go` (which is part of the `orchestrator` package) will not introduce a circular dependency.
- **Test Alignment**: `internal/orchestrator/orchestrator_test.go` already expects the correct counts (13 for Standard, 14 for Deep).

## 4. Technical Constraints
- The `orchestrator` package depends on `scanner`.
- `scanner` MUST NOT depend on `orchestrator`.

## 5. Architecture Documentation
- The project uses a modular "Phase" approach for scans.
- Vulnerability verification is centralized in `VerificationEngine`, with deterministic fallbacks living in the `scanner` package.
- AI-driven attacks are coordinated by specialized executors in the `attack` package.
